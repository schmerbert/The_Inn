# REVIEW_TEMPLATE.md

## Purpose

Use this template to review a marble without forcing it into a standard shape.

Review should protect invariants without freezing local form.

A review is not an aesthetic judgment. It asks whether the marble is inheritable, operable, and safe enough for its own work.

## Review header

```text
Marble:
Reviewer:
Date:
Version / commit / state:
Last real use:
Review type: first-pass / second-pass / daily-driver / safety / pre-public / maintenance
```

## 1. Doorway clarity

Does the marble clearly explain what it is for?

- What recurring work does it hold?
- Who is the model here?
- Where is the model allowed to stand?
- What role must it not inflate into?
- Does it distinguish itself from a prompt, persona, or generic scaffold?

## 6. Manual breath

Manual breath should be operable before automation.

- Can it place material?
- Can it label authority?
- Can it retrieve a fitted return?
- Can it silence stale or irrelevant material?
- Is automatic breath absent, off, or clearly a refactor of manual breath?

## 7. Tests

A first pass needs both a hostile test and a positive path.

Hostile test:

- Does it try the real forbidden cell?
- Is the bad attempt tempting, not cartoonishly bad?
- Does it touch the same path real work uses?
- Is refusal recorded?

Positive path:

- Does a valid record pass?
- Does it land in the correct surface?
- Does it receive the correct authority label?
- Are remaining limits recorded?

## 9. Friction classification

What friction appeared?

Productive friction protects the law. Accidental friction blocks use.

```text
Productive friction to preserve:

Accidental friction to patch:

Unclear friction to test again:
```

Questions:

- Is the marble heavy in a useful place?
- Is it missing a surface in an ordinary path?
- Did the model have to improvise storage?
- Did a name fail to help action?

**Strain still returning after honest patches?** Check whether the pressure matches a named pattern in [`ADVANCED_MARBLES.md`](ADVANCED_MARBLES.md) §1. If borrowing a pattern, answer the Gate questions (§ The Gate) before building. Record adopted and declined patterns in handoff or `IDEAS.md`.

## 12. Horizon control

Has the marble built horizon features too early?

**Correct use of the friction shelf:** `ADVANCED_MARBLES.md` is for recurring strain on a working marble — lookup via §1, one pattern at a time, Gate questions answered. Incorrect use: reading cover to cover, treating the table of contents as a feature menu, or adding forest/pond/window/imp before the floor holds.

Check:

- Rooms exist only if they change context, tools, memory, or permissions.
- Automation exists only if manual breath was trusted first.
- Validators are small and refactor manual practice.
- Template/specimen patterns did not override local fit.
