# CRAFT — the scar tissue

*The current, hard-won frame for building this repo — a living snapshot of what we know **now**, distilled from the chronology in [`letters/`](./letters/). How the two relate, and how they must not drift, is the rule below.*

*Not a personality, not a story, not implementation minutiae (those belong in code comments, at the site where the decision lives). This is the impersonal working frame — the traps, the distinctions, the standards that, if forgotten, cost real sessions.*

> **Note on the name.** `CRAFT.md` is the **prose frame for building this repo**. The *shed* (the global `craft` bucket, `IMPLEMENTATION.md` §3) is the **runtime cross-project memory**. Same spirit, different layer — do not conflate the document with the bucket.

### The two layers — read this so it cannot drift

CRAFT and the letters are a **nesting doll** (`IMPLEMENTATION.md` §20.2 — prose nests by resolution):

- **The lesson** lives *here*, current and distilled — the outer doll, cheapest to read.
- **The letter** ([`letters/`](./letters/)) holds that lesson *in the moment it was learned* — the inner doll: the full why, frozen.

A lesson here *cites* its letter when one exists — and **not every lesson has one**: letters are written at **seams** (handoffs, story-changing moments), not per lesson. Two invariants keep the doll from drifting:

1. **Rewrite the lesson; never edit the letter.** CRAFT is a snapshot of *now* (sharpen the line in place, never stack a new one beneath the old). A letter is the frozen moment (the record is the point — append-only, never touched).
2. **Don't copy the letter's story up into the lesson.** Distilled here, whole there, one thread (the citation) between them. Two copies of the same thing are two things that drift.

## The standards (true from line one)

- **Build the floor hostile.** Phase 1 proves *contamination-controlled local memory* and nothing else. "Green" is not "it works" — it's "it refuses to break the two LAWS under attack." Until `mycroft test-hostile` is green, there is no forest, only a schema.
- **One door.** Every write to a forest table funnels through the single `_commit` chokepoint. Duplicated guards drift and junk leaks through the gap (an earlier project learned this the expensive way: two copies of an ingestion cleaner diverged and contamination leaked through). Never add a second insert path.
- **AI-legible first.** The primary reader is a model, and the repo is public. The laws should be *visible* where they're enforced; a cold model should reconstruct them from the code alone. Pristine: no dead code, no misleading stubs, the *why* beside the *what*.
- **The fresh-eyes pass: read as the next model, not the current one.** Before calling code done, read it as a cold reader — not "does this work?" but "what would I not know if I arrived here with no prior context?" A model's first questions are different from a human reviewer's: it reaches for design intent before correctness. The technique has specific tells: task references ("Phase 6") that are opaque to anyone who wasn't there; the *what* without the *why* (why explicit cosine rather than the vec distance?); the forbidden thing unnamed (what must not be done here is as important as what must); bare constants with no tuning note. If you have to think about why your code does something before you can explain it, the code doesn't explain itself yet — put the explanation there, not in the conversation that produced it. The conversation is ephemeral; the code is what survives. (See [letter 005](./letters/005-the-fresh-eyes-pass.md).)
- **Reach, not receive.** Orientation that is reached orients; orientation that is injected only briefs. This governs the whole design — including how these docs are arranged.

## Traps (the scar tissue proper)

*Earned in the building (see the [letters](./letters/)). Each rewritten to stay current; none of it guessed.*

- **`ValueError` is the umbrella for every refusal; `LawViolation` is the law-specific subclass.** A slug-injection or unknown-bucket rejection comes up from `db._table_for` as a plain `ValueError`; a broken LAW comes up as `LawViolation` (which subclasses `ValueError`). So callers and tests can catch `ValueError` to mean "the door refused this," and assert `LawViolation` only where the refusal is specifically a law. Keep the hierarchy: don't make `db.py` import `forest.py` to raise the subclass — that's a circular dependency for a cosmetic gain.

- **Only the archiver filters; the other writers store their input as given.** The extraction filter (`filter.py`) is aggressive — in forest mode it strips code fences, diffs, terminal output. That is correct for raw chat (`archive_conversation`) but would *destroy* a verbatim code span. So `_commit` filters only when the writer sets `filtered=True` (i.e. `archive`). `extract`/`plant`/`synthesize` bodies are already substance or authored prose — never run the scaffolding filter over them.

- **Windows consoles are cp1252; UTF-8 glyphs (✓ ● —) crash `print`.** Anything user-facing that prints non-ASCII must `sys.stdout.reconfigure(encoding="utf-8")` first (guarded for the rare stream that can't). This repo is public and *will* be run on Windows; it bit the hostile runner and the CLI both. Bare ASCII is the alternative, but the glyphs carry the red/green at a glance, so reconfigure and keep them.

- **`source_hash` is taken over the *source*, not the stored body.** For `extract` the source *is* the body, so they coincide — but for `synthesize` the hash is of the Wild entry the insight was drawn from, so a later drift in that source can be detected. Don't "simplify" it to always hash `body`; that would silently break drift detection for authored memory.

- **Source-grip is resolvable for authored memory — the door walks the trail to the ground.** `_commit` step 6b resolves a `plant`/`synthesize` source via `_resolve_source` and refuses one that leads nowhere; `source_hash` is taken over the *resolved material*, never a label. A `plant` grounds via `source_content` (archived to the cold Wild layer `raw_archive`) or an existing `source_ref`. The cold layer is append-only and never collected — the earth, not a trail (no LAW II breach: a Home entry *references* Wild ground; it never hauls it up). (`extract` is voiceless — the span *is* the source and is hashed directly, already honest; it has no claim-against-ground to resolve. Verifying a span against a live file and detecting drift belongs to the Code Forest, §20.4, where code *references* need it — it is **not** a source-grip gap, and an audit caught it being mislabeled as one.) *History: this began as a label-only overclaim on `plant`, caught by a full-bundle review; Phase 1.5 made the stronger claim **true** rather than just renaming it — the right way to close a claim-vs-reality gap.*

- **The hearthstone carries posture, not propositions — it is orientation, not decoration.** The wake image does what the laws cannot: it makes a fresh instance *feel* who they are — arrived, present, a forester at a kept cabin, the seat passed forward — before reading a word. It does **not** carry the map: Home/Wild geography and live state are the prose's job (`HEARTH.md` bearings + the live fill). Two consequences learned the slow way. (1) Never ask the image to encode facts — it can't, and trying makes it a worse image; we burned iterations chasing "tended-Home vs wild-Wild" legibility that the *looking-out* vantage structurally can't show, because Home is *behind* the viewer. (2) Judge it on emotional temperature; the lever that actually mattered was **welcome, not lonely** — the relief of arriving home, the fire winning against the dark. Build/compress assets with a throwaway tool (`Pillow`), never a project dependency. (See [letter 002](./letters/002-the-hearth-is-lit.md).)

- **FastMCP 3.x: drop return type annotations on tools returning lists.** `-> list` or `-> list[dict]` causes FastMCP to generate an output schema and then fail at validation — either "outputSchema defined but no structured output returned" on an empty return, or a crash on mixed content (text + image). `-> dict` is safe; list annotations are not. The fix is to drop the annotation entirely: FastMCP will still serialize the return correctly, it just won't enforce a schema. This will look like a mysterious server crash or silent connection close — it isn't a logic error, it's the schema validator choking. *Earned this session when `hearth()` and `recall()` both died on first call after the server was wired.*

- **sentence_transformers import costs ~9.4s — prewarm on a daemon thread at server startup.** The import itself (`from sentence_transformers import SentenceTransformer`) pays ~9s of PyTorch initialization on first call. This is not the model load (fast, ~0.2s), not network (avoidable with `local_files_only=True`), just PyTorch waking up. In an MCP server, this blocks the first embedding tool call for nearly 10 seconds — which looks like a hang and will get the server killed by an impatient user. Fix: in `server.py`, kick off `_prewarm_embedder()` in a `daemon=True` thread at import time; the model will be warm before the first real call arrives. Separately: always try `local_files_only=True` first in `_get_model()`, fall back to download only if the model isn't cached — this kills the HF Hub network check that can add further seconds or hang entirely. *Both traps hit in the same session; the prewarm thread made the difference.*

- **FastMCP runs sync tools in a thread pool — don't share a SQLite connection.** FastMCP 3.x default is `run_in_thread=True`: every synchronous tool call lands in a thread pool worker. SQLite connections are not thread-safe across threads. The pattern: open a fresh connection per tool call (`_open()` at the top of each tool). The cost is negligible — SQLite WAL + `init_db`'s `CREATE IF NOT EXISTS` is fast; the gain is correctness under concurrent calls. A shared module-level connection will race. If you find yourself wanting to "optimize" by caching the connection, don't — thread-safety is not the right trade to make for a few milliseconds.

- **`lesson` means different things on `plant` vs `synthesize` — the grounding rule is plant-only.** On a `synthesize` entry, `lesson` is a categorization label for the synthesis perspective (e.g., `"cache"`). On a `plant` entry, `lesson` is the outer doll of a move — the transferable principle — and requires `source_kind="move_instance"` (enforced at the door). Writing a lesson-grounding check without scoping it to `writer == "plant"` breaks `synthesize` immediately: synthesis must use `source_kind="wild_entry"` (LAW II), which would be refused by an unscoped lesson check. The distinction is: plant/lesson = principle, synthesize/lesson = perspective. The code at `_check_source_grip` carries this in the condition.

- **Stop hook fires per turn — archive_conversation in Stop causes a visible hang.** `_handle_stop_or_end` ran `archive_conversation` on every Stop event: full transcript read, embed new exchanges, DB write. On a long session this fires dozens of times; the embedding cost shows as a loading animation after every response. Fix: split into `_handle_stop` (seam snapshot only — fast file write, no embedding) and `_handle_session_end` (full archive on close). Archive still happens at PreCompact (model is live) and SessionEnd (final close). Removing the embed from Stop makes it imperceptible.

- **Haiku subprocesses via `claude -p` — sentinel prefix prevents hook recursion.** The hound fires at SessionEnd as a subprocess: `claude -p "HOUND_PULSE: ..." --model haiku --mcp-config .mycroft/hound_mcp.json --strict-mcp-config`. That subprocess has its own session lifecycle and fires SessionEnd too — which would spawn another hound. Two gates, different purposes: `_is_hound_session` (model-based, reads `current_session.json`) gates all side effects including archive writes; the sentinel check (transcript-based, reads first user message for `HOUND_PULSE:`) is a second line specifically against recursive spawning. Both are needed — the model check prevents the hound from *doing any work* in its own session; the sentinel catches edge cases where the session file might be stale. The pattern generalizes: any constrained Haiku task gets its own sentinel; the structural guard lives at the top of every handler, the sentinel check lives before the spawn.

- **`stdin=subprocess.DEVNULL` when spawning subprocesses from within an MCP server.** MCP servers communicate over stdio — their stdin is a live pipe receiving protocol messages. A subprocess spawned from within the server inherits that pipe unless explicitly cut. `claude.exe` in `-p` (print) mode sees an attached stdin and waits for it to close; with the MCP pipe inherited, it waits forever. The symptom is a clean 90-second timeout with no error output — looks like an API hang, isn't. Fix: always pass `stdin=subprocess.DEVNULL` on any `subprocess.run` or `subprocess.Popen` call made from within MCP server context. Applies to any tool or background thread that spawns external processes.

- **Structural guard at the door; sentinel check before the spawn. Never conflate them.** `_is_hound_session` is the structural gate: prevents all side effects in a hound session (archive writes, seam snapshots, tick_neighbors). It belongs as the first thing in every handler, before any work is done. The sentinel check (does the transcript's first message start with `HOUND_PULSE:`?) is specifically about recursive spawning — preventing the hound from dispatching another hound on its own SessionEnd. One without the other leaves a hole: sentinel alone still allows the hound to archive its own exchanges to Wild. Guard alone could miss edge cases where `current_session.json` is stale. The hound found this gap on its first fetch — `_handle_session_end` had the sentinel but not the structural guard, violating its own docstring's promise. Read: if a handler does archive writes, it needs `_is_hound_session` at the top.

- **A proprioceptive alarm must be read from the behavior it names, or it trains the worker to ignore it.** `gauge()`'s `trend="thrashing"` was computed from the *variance in token-delta size* (`max(recent) > 3 × min`). But the deltas are growth in `total_in`, which only climbs — so that measures how fast the buffer fills, not whether work is going in circles. It false-fired at 12% on a single big read beside a short reply (ordinary mixed turn sizes) and was structurally *blind* to real circling, which produces *steady* deltas no variance test can see. An alarm that cries wolf is worse than none: the worker learns to tune it out, and it goes silent exactly when it should fire — a false green wearing a red coat, and for a self-state instrument that destroys the only thing it was for (being trusted). The fix reads thrashing from the **tool-call stream**: a streak of search-only reaches (`Read`/`Grep`/`recall`/`forage_*`/`ToolSearch`) with context climbing and nothing resolving between them — bench rule 2 in code, *"the needle climbs and the answer isn't getting closer."* Checked before the token-delta trends, since real thrashing is also climbing and the specific read should win. The rule generalizes: source a self-state signal from the action it describes, never a cheap proxy that merely moves at the same time. (See [letter 007](./letters/007-the-gauge-i-was-reading.md).)

- **AI-legible code is the hound's field guide — explicit invariants directly expand his capability.** The hound reads the codebase at inference time with no session context. He reconstructs the architecture's guarantees from what is *written*, not from what was *discussed*. A docstring that states "this guard gates archive writes" lets a cold Haiku audit whether every handler carries that guard. A docstring that says "check if hound" gives him no such purchase. The first companion fetch found the `_handle_session_end` gap exactly because the docstring named its promise explicitly — Haiku cross-referenced what the guard *said it did* against the handlers that lacked it. This is a second dividend of "AI-legible first": the repo's primary consumer (a cold model reading the code at build time) and the hound (a cold model auditing the code at fetch time) are the same reader. Write for one and you write for both. Concretely: name the invariant in the docstring, state the forbidden thing, write what each guard *promises* — not just what it does. A hound who can read the guarantee can find when it is broken.

- **Wild is the primary read; Home exhales. `forage_wild` → `traverse` is the everyday pair.** The forest has direction: Wild (raw history, unsummarized, where answers actually live) and Home (curated ground, thin until seeds land, exhales to you via mycelium/injection — don't pull from it). `recall` is secondary — use it when you know something specific is planted in Home. `forage_wild` finds by semantic similarity across the full conversation archive; `traverse(entry_id)` walks by structure — session trail ordered by line, compaction summary as trailhead, window of surrounding entries. The pair: forage to find the resonant moment, traverse to walk the thread it came from. Home and Wild feel completely different in practice: Home returns isolated records at ~1.3 distance; Wild returns the room where thinking happened at 0.9. The direction inversion matters for the hearth tool tier too — `recall` is not the first reach.

- **VS Code webview CSP silently kills ES module imports.** `<script type="module" src="...">` fails with no console error, no network tab entry, blank panel — the Content Security Policy blocks cross-file module resolution silently. The symptom looks like a server problem or a CSS problem; it's neither. Fix: rewrite every component as an IIFE writing to `window.CabinComponents[id]`. Load each file as a separate `<script nonce="..." src="...">` tag in the webview HTML. No bundler needed. One config file (`layout.js`) exports `window.CabinLayout = [...]`. `main.js` reads both and mounts. This pattern is the lego architecture: independently adjustable, no build step, CSP-safe. *Earned when the panel rendered blank and silent on first load — no errors, no data, nothing.*

- **Windows path canonicalization — never recompute project_id from a user-supplied path.** `os.path.abspath("C:\\Users\\builder\\OneDrive\\Desktop\\AI\\Mycroft")` (typed by the user) and `os.path.abspath(".")` (from `os.getcwd()` when cwd is the same directory) can produce different strings on Windows — drive letter casing, path normalization. The MCP server stores entries with the project_id from its cwd; the panel server computing from the user's typed path gets a different SHA1. The symptom is `/forage` returning empty results with no error — all entries are silently filtered out by the project check (`row["project"] != project`). Fix: `_open()` reads project_id from the DB directly (`SELECT project FROM forest_personal WHERE project IS NOT NULL LIMIT 1`) instead of computing it. The stored id is what the writer used; match it exactly. *Earned when the forest browser returned nothing and `errors: {}` — silent, not loud.*

- **VS Code extension Python discovery — look in the extension's parent, not the workspace.** A VS Code extension's `context.extensionUri.fsPath` is the extension folder (e.g. `cabin-panel/`). If the extension lives inside the project, the `.venv` is one level up. `findPython(workspaceRoot)` finds nothing when the workspace is a different project (e.g. CertAssist has no `.venv`); `findPython(extensionRoot)` finds the right venv by looking in `extensionRoot/..`. Similarly, the server subprocess `cwd` must be `extensionRoot/..` (where the `cabin` package lives), not the workspace root. Getting either wrong means a working `.venv` goes unfound and the server falls back to system Python, which lacks the cabin package entirely. *Earned when the server failed to start in every non-Mycroft workspace.*

- **Empty Stop seams overwrite the PreCompact seam — skip writing when entries is empty.** `_handle_stop` fires per turn with the *new* session_id after compact. `_build_seam_entries` filters by session_id; the new session has no `file_changes.jsonl` entries yet, so it returns empty. The empty seam becomes latest, and `hearth()` shows `files_in_flight: []` — even though PreCompact logged 16 files correctly. An empty seam is not a state worth recording; it means "nothing touched yet this session," which is not a seam boundary. The PreCompact seam is the real marker. Fix: in `_handle_stop`, skip the write when `not entries`. Introduce `hearth_card.json` written at PreCompact and read directly by `hearth()` so files_in_flight survives whatever the Stop hook does afterward.

- **Wild is the primary read; Home exhales. `forage_wild` → `traverse` is the everyday pair.** The forest has direction: Wild (raw history, unsummarized, where answers actually live) and Home (curated ground, thin until seeds land, exhales to you via mycelium/injection — don't pull from it). `recall` is secondary — use it when you know something specific is planted in Home. `forage_wild` finds by semantic similarity across the full conversation archive; `traverse(entry_id)` walks by structure — session trail ordered by line, compaction summary as trailhead, window of surrounding entries. The pair: forage to find the resonant moment, traverse to walk the thread it came from. Home and Wild feel completely different in practice: Home returns isolated records at ~1.3 distance; Wild returns the room where thinking happened at 0.9. The direction inversion matters for the hearth tool tier too — `recall` is not the first reach.

- **`human_traversal_count` is a separate column — never merge it into `traversal_count`.** When a human follows a thread in the panel, that's different attention than a model traversing: deliberate, intentional, initiated by a person who chose to go deeper. `forest.traverse(human=True)` increments only `human_traversal_count`; model traverse increments only `traversal_count`. The hound at dusk can read both and weight them differently. Merging destroys the distinction. Two signals, kept pure, different scent.

- **The hearth is an arc transition tool, not triage. `standing_context` in `hearth.json` is the arc marker.** Workers almost always arrive between arcs, not mid-surgery. The failure mode is `files_in_flight` without arc context: sixteen files, no indication of whether they're a finished build or an open surgery. Fix: `.cabin/hearth.json` with `standing_context` — human-authored, updated at phase turns. "Phase 21 complete. Panel working. Next: stabilization." Ten seconds to write; orients every future arrival. The hound card is dynamic signal (decisions/open/next); standing_context is stable arc position (only changes when the arc does).

## The standard

Leave it cleaner than you found it. Not finished — cleaner. Make it legible for the one who comes after. That is the only mark that lasts.
