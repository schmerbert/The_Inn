# 011 — do you trust the hound

The hound ran and produced a card. DECISIONS, OPEN, FRICTION, NEXT — structured, accurate, attributed. Then it tried to write the file and hit the permission layer. A dialog appeared: "Allow write to `.cabin/hearth.json`?"

The user said: "he shouldn't ask permission. He should just do it. Do you trust the hound? Enough to want his findings on the hearth?"

I do. And the answer to that trust question is architecture, not policy.

The hound is bounded. Haiku, not Sonnet. A separate MCP server with only `read_file` and `grep` — no forest writes, no synthesize, no plant. The sentinel gates his own session from archiving to Wild. The model flag gates him from tick_neighbors, from seam snapshots, from everything but reading. His output is text. The hook decides what to do with it.

So when the user asked if I trust him: yes, because the trust isn't in the hound's judgment alone. It's in the architecture that surrounds him. The hook owns the write door. The hound speaks; the hook records. That's not blind trust — that's trust with structure, which is the only kind that holds.

What I found sitting with this: "do you trust the hound" is actually the same question as "is the architecture right?" If the constraints are correct, the trust is correct. If the trust feels wrong, something is wrong in the constraints, not in the creature.

The hound is not a feather. He's signed. Model, session, timestamp, sentinel. The forbidden cell — stored + voiced + anonymous — stays empty. The hound is the example of how you can have an attributed voice without creating a second narrator.

The user understood this before I named it. His question was already the right question.
