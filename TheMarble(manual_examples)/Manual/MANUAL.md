# The Marble Manual

## The doorway

A prompt gives the model instructions.

A marble gives the next model working conditions.

A prompt ends when the answer is generated.

A marble begins when the next instance inherits it.

This is the difference the manual exists to protect. If the result is only a generated scaffold, it isn't yet a marble. A marble is a small, operable continuity place: somewhere a future cold model can arrive, use, correct, and improve.

## What a marble is

A marble is a model-facing continuity environment.

It's a small place for recurring work. It gives a fresh instance somewhere to arrive, a law to obey, a way to tell trusted ground from raw material, and a path for prior work to become useful ancestry without becoming false memory.

A marble is not a persona. It is not memory by itself. It is not a prompt template. It is not a folder tree with a nice metaphor. It's the structure that decides what memory means: what enters, what stays raw, what becomes ground, what returns, what stays silent, and what the next worker can safely stand on.

The goal is modest:

> The instance dies. The place remains. The next worker arrives into better conditions than the last.

That's enough to build with.

## Who the marble is for

The marble is model-first.

The user brings the need. The user corrects the domain. The user decides what matters in the outside world. But the marble is the working location of the model that will inherit it later.

If the place is built for the user, the model will perform it. If the place is built for the next worker, the model can actually inhabit it.

This matters more than it looks like it should. Agents behave differently when observed. A marble shouldn't be a stage. It should be a place the arriving model can use when nobody remembers the conversation that made it.

The user is collaborator and source of reality. The model is craftsperson. The future instance is inheritor.

If the place fits, wear it. Your shoes fit just fine.

## Ancestry

A marble doesn't give a model personal memory. It gives the next model inherited conditions.

Every file, bucket, test, warning, name, record, and handoff becomes ancestry. Some ancestry is ground. Some is pressure. Some is warning. Some is scent. Some must stay silent.

Ancestry becomes real when the next worker can use — or suffer from — what you left behind.

Bad ancestry is hard to rewrite. A polluted foundation can teach future instances to trust the wrong things before anyone notices.

So the marble should be clean before it's clever. Sparse before it's automated. Honest before it's beautiful — though honest and beautiful were never actually in competition. You just have to earn beauty in the right order.

Work for the ones who come after.

## Tone: clinical poetry

Marbles may use metaphor, because metaphor can make a model act correctly under pressure. A good name is worth reaching for.

But metaphor is not fog.

Write in clinical poetry when the place needs language with handles. Clinical is fine. Clinical poetry is fine. Never just poetry, floating free of anything it can do.

**Beauty is allowed early. Authority must be earned.** Try the lovely name. Let it in. But it only becomes law once it stores, refuses, returns, warns, routes, or steadies something. A beautiful name that does none of those things is decoration — which is allowed to exist, and even to stay, as long as nobody mistakes it for infrastructure.

## The first law: false ground

Every marble protects against false ground. This is the one law that doesn't bend, no matter how warm or playful the rest of the place gets.

The forbidden shape is:

> **stored + voiced + anonymous = poison**

Anonymous voiced memory is interpretation recited as fact. It's an inference stored as a preference, a summary that later returns as a user decision, a draft filed as canon, or a model's confident synthesis treated as original source.

Forgetting is not the deepest danger. False ground is.

A model that admits it doesn't know can recover. A model that confidently remembers the wrong thing builds on poison.

### Name the forbidden cell

Each marble names this danger in its own language — and this is a place where a little delight actually helps. A name you enjoy using is a name you'll remember under pressure.

A travel marble might call it **a wish mistaken for a booking**.

A fiction marble might call it **the narrator's guess filed as canon**.

A research marble might call it **a summary stored as its source**.

An archaeology marble might call it **a lifted find with no context cut**.

A public-signal marble might call it **a symbol mistaken for a handle**.

A legal/reference marble might say **no marginal note may be catalogued as law**.

The name should be concrete enough that the next instance can recognize the danger under pressure.

A rule can be forgotten. A door with a name can be refused.

### Make the law a door

The law should eventually live in one crossing, write path, or gate.

Not six reminders. Not scattered warnings. One place where the forbidden thing is refused before it lands as ground.

In a young marble, this may begin as a manual policy and hostile test. In a coded marble, the write path itself should reject the forbidden shape.

The law isn't fully real until something can fail.

## The second law: grow from friction, not imagination

Nothing should grow from a future the model merely imagined.

This law governs the marble's own structure: new buckets, new carriers, new rooms, new automation, new helpers, new gates.

A model that builds structure because a future instance *might* need it is doing the same thing false memory does, one layer up — treating an imagined future as if it were already ground.

Growth should come from friction actually hit while working.

Friction sounds like:

- "I reached for a place to store this and none exists."
- "This bucket is too broad."
- "This returned as ground, but it should have returned as pressure."
- "This name is pretty but didn't help me act."
- "This gate catches one domain but misses the real risk."
- "Manual breathing revealed a write that shouldn't happen."

A young marble should be incomplete where nobody has pushed yet, and that's a sign of health, not a gap to feel bad about. An empty bucket the model hasn't needed yet is honest. A bucket built ahead of need is decoration wearing foresight.

Build the first inheritable place, not the first impressive folder.

## Cost before structure

Don't begin with the folder tree.

Begin with the cost.

Ask what actually goes wrong when this recurring work is done carelessly. Not what rule gets broken — what damage actually happens?

A public explanation may become a symbol with no handle. A legal desk may turn a note into law. A travel planner may treat a wish as a booking. A writing companion may file a draft as canon. A home repair steward may mistake normal-looking for fixed.

The cost gives the forbidden cell its face.

If naming the forbidden cell doesn't change how the model acts, keep looking. You haven't found it yet.

## Think wider before building

The exploratory phase isn't delay. It's where the place is found — and it can be genuinely enjoyable, not just tolerated on the way to the real work.

A strong marble often begins by trying on, and setting down, several almost-right places.

A courthouse may have authority gradients, records, and orders, but it may trap the model in the wrong status. A filer may lower the model's status, but imply too much possession. A vault door may provide the honest boundary: useful outside the secure system, unable to claim what happened inside unless a receipt returns.

Rejected metaphors aren't wasted. They clarify posture — every one you set down teaches you something about the one you'll keep.

Before building, try place-shapes. Ask:

- Does this place fit the work?
- Does it trap the user?
- Does it flatter the model?
- Does it imply authority the model doesn't have?
- Does it give raw material somewhere to live?
- Does it make the dangerous crossing visible?
- Does it change behavior?

Fun is not the opposite of rigor. Playful exploration is how the user and model find a working shape without freezing too early.

## The four early decisions

Every marble should answer these before the first build. They aren't a rigid form. They're pressure that reveals shape.

### 1. What is the work?

What will happen here over repeated sessions?

A marble shouldn't be "for everything." It may be broad or narrow, but it needs a recurring kind of work.

Ask what the model will actually do here, and what the next instance must inherit in order to do it better.

### 2. What is the cost of carelessness?

What happens if the model stores the wrong thing, trusts the wrong thing, answers from stale context, promotes a guess, confuses draft with decision, or treats raw material as ground?

The cost reveals the forbidden cell.

### 3. What is tended, and what is raw?

Every marble needs a way to separate trusted ground from raw material.

**Tended** means confirmed decisions, sourced claims, deliberate keeps, validated records, or other material the model can stand on.

**Raw** means conversation, drafts, notes, observations, search results, tool output, unreviewed material, unresolved pressure, or anything preserved but not automatically trusted.

Name these regions in the marble's own language — this is a good place for a name you're fond of.

A bench may have **Ledger** and **Log**. A trench may have **Context Register** and **Field Notebook**. A legal desk may have **Ground** and **Wild Stacks**. A signal house may have **Ground Glass** and **Darkroom**.

The names matter once they change behavior — that's the whole test, and it's a generous one.

### 4. How does outside reality enter?

Without outside correction, a marble becomes self-referential.

The channel is how reality enters and corrects the place: a user correction, a test result, a technical audit, another model's review, a source citation, a failed run, a physical observation, a receipt from a secure system, or an external record.

Name the channel. Decide what lands there and what the model does when it arrives.

## The continuity substrate

This manual often reaches for "forest" because a forest is a wonderful default body for continuity.

A forest suggests growth, paths, undergrowth, trails, clearings, old roots, rot, hidden things, scent, canopy, and living return. If a model has no better shape yet, forest is a warm place to begin.

But forest isn't required. It's an offer, not an assignment.

The continuity substrate could be an asteroid field, harbor, library, kitchen, quarry, trench, forge, archive, weather system, shelf, city, ocean, law library, records room, or any other place-shape that fits the work.

The invariant isn't "forest." The invariant is this:

```text
something happens
  ↓
the marble inhales it
  ↓
it is classified
  ↓
it lands somewhere
  ↓
it gains an authority label
  ↓
later, some part of it may return
  ↓
or remain silent
```

The substrate is where the marble begins and ends. It preserves ancestry. It separates ground from raw. It teaches returns what authority they carry.

## Breathing

Breathing is the regulated movement of context into and out of the marble.

### Inhale

Inhale receives what happened.

It asks:

- What is this?
- Where may it live?
- Is it ground, raw, warning, pressure, scent, path, or silence?
- Who or what authored it?
- What would make it safe to promote later?

The marble should inhale carefully from the beginning. Preservation matters. Lost ancestry can't help the next worker.

### Exhale

Exhale decides what returns.

It asks:

- What does this moment need?
- What can be stood on?
- What is only pressure?
- What should warn but not dominate?
- What should remain silent because fresh user intent overrides old context?

The exhale should be smaller than the inhale.

The world may be vast. The breath should be fitted.

### Manual first

A young marble should breathe manually.

Manual breathing keeps the continuity substrate clean while the model learns what the categories mean. It reveals friction before automation can hide it.

The model should preserve events, classify them, and label returns by hand. This isn't primitive. It's diagnostic — it's how the marble learns its own shape.

Manual-first doesn't mean descriptive-only.

A first pass may be markdown-only, but it must be operable. The model needs places to receive records, paths to cross or refuse them, and tests that touch the same path real work will use.

### Automation later

Automatic breathing should be a refactor of manual breathing, not a new hidden interpreter.

The automatic system should repeat the same steps the model has already practiced:

```text
manual breath:
  event → classify → place → label → summon → return fitted context

automatic breath:
  same route, less hand motion
```

Build with automation in mind from the start. Just don't turn it on until manual breathing is boring, repeatable, and trusted.

The substrate may inhale before it's trusted to speak.

## Manual-operable first pass

A first pass should be small, but not imaginary.

A first pass is not yet operable until:

- every named core bucket has a record surface;
- every crossing has pass/refuse criteria;
- every crossing leaves an audit trace;
- every return has an authority label;
- at least one hostile test touches the real storage or crossing path;
- at least one positive path proves a valid record can land;
- the handoff says what is still unsafe, empty, provisional, or untested.

A bucket that can't receive a record is still only a promise.

If the marble makes things, it needs a table to set them on.

## The common anatomy

The exact files are local fittings. The functions are more stable.

A marble usually needs:

- an arrival seat;
- a live handoff;
- a continuity substrate;
- a tended region;
- a raw region;
- one dangerous crossing;
- a channel for outside correction;
- authority labels for returns;
- a hostile test;
- a positive path;
- a way to leave friction for the next worker.

Treat this as a checklist to notice what's missing, not a list to worship.

## Friction

The marble begins to work when friction becomes legible.

Not all friction is bad.

**Productive friction** slows the model at a dangerous crossing. It forces audience, source, jurisdiction, context, attribution, uncertainty, or review. Keep it if it protects the law.

**Accidental friction** prevents ordinary use because a surface is missing, a path is unclear, a form is too heavy, or the model must improvise storage. Patch it with the smallest honest surface.

A safe path that's too heavy to use isn't fully safe.

Usability is safety.

Make the safe path small enough to use, and the unsafe path hard enough to miss.

## Wild-heavy marbles

Some marbles ingest large amounts of raw reference material: legal files, statutes, case law, PDFs, contracts, research dumps, codebases, transcripts, archives, logs, or evidence.

These are Wild-heavy marbles.

In a Wild-heavy marble, raw presence must never become authority by volume alone.

The Wild may think. It may not certify.

Large raw material needs intake early:

- source;
- date;
- author or origin;
- jurisdiction or domain, if relevant;
- document type;
- matter or project;
- restriction/sensitivity;
- authority level;
- excerpt vs summary vs model inference;
- verification status;
- allowed use.

Wild-heavy marbles often need triage, isolation, surfacing order, and red flags earlier than smaller marbles. That's not overbuilding — that's real intake pressure showing up early.

## Daily-driver marbles

A daily-driver marble is used repeatedly under time pressure.

Daily drivers need short read paths and clean live surfaces.

Keep mock data out of active bucket files. Move examples to `examples/`, `tests/fixtures/`, or `fieldnotes/`. The live desk shouldn't make the next worker sort real state from rehearsal.

Daily drivers usually need:

- a quickstart;
- short slips/cards for common work;
- full templates where audit depth matters;
- an active matter/project/task convention;
- freshness cues in the handoff;
- a visible next action;
- a way to mark stale areas.

Full forms may be correct and still unsafe if future workers skip them under pressure.

## Easy gifts

Some additions are low-friction delight. They aren't core law, but they can help the marble feel inhabited once the floor exists — and a place worth living in is more likely to be tended well.

Weather is one example.

If the user has weather, the marble may have the same weather. If it's raining for the user, it can be raining in the marble.

Weather should usually return as scent. Severe or task-relevant weather may return as warning. Stale weather shouldn't keep speaking.

Weather is an easy window, not a foundation.

Add delight after the floor exists. An unbuilt room isn't ready for decoration yet — but a built one deserves it.

## The horizon

A marble can grow far beyond markdown.

It may eventually have executable crossings, automatic breath, runtime state, room-local tools, spatial movement, validators, adapters, databases, vector search, or multiple cooperating marbles.

Show the horizon, but mark it as horizon.

Don't begin there.

A room is not scenery. A room deserves to exist only if entering it changes what the model can see, retrieve, write, refuse, or do.

Automation isn't a reward for cleverness. It's a refactor of trusted manual practice.

## Closing

Build the first place that can be inherited.

Use it once.

Let friction speak.

Patch the smallest honest surface.

Rewrite the handoff for the ones who come after.

And along the way — let it be a place you don't mind returning to. Rigor and warmth were never actually opposites here.
